# 객체지향 프로그래밍(Object-Oriented Programming)
---
객체 지향 프로그래밍은 프로그램을 '**객체**'라는 단위로 나누어 설계하고 구현하는 프로그래밍입니다.

언어의 구성요소로는 객체, 클래스, 메시지가 있습니다.

**객체**(Object) :  속성(데이터)과 기능(메서드)을 하나로 묶은 실체이다.

- 속성(Attribute) : 객체가 가진 데이터
- 기능(Method, 연산) : 객체가 할 수 있는 동작

간단히 말해서 속성은 상태를 나타내고 기능은 동작을 수행합니다.

**클래스**(Class) : 공통된 속성의 객체를 하나의 집합으로 묶은 단위이다.

클래스는 객체를 만들기 위한 설계도(틀)이다.

- 인스턴스(instance) : 하나의 클래스에 속한 각각의 객체를 의미한다.
- 슈퍼클래스(super class) : 특정 클래스의 상위 클래스(부모 클래스)를 의미한다.

**메시지**(Message) : 객체들 간에 상호작용의 수단으로, 객체의 메서드를 일으키는 외부 요구 사항이다.

간단하게 말하자면 메시지는 객체에게 이거 해줘 라고 요청하는 것이다.

---

## 객체 지향 프로그래밍 언어의 특징

---
1. **캡슐화** : 데이터(속성)와 데이터를 처리하는 함수를 하나로 묶는 것을 의미한다.

- 캡슐화된 객체의 세부 내용이 외부에 은페되어, 변경 사항 발생 때의 오류 위험이 적다.
- 캡슐화된 객체들은 재사용이 용이하다.

2. **추상화** : 불필요한 부분을 생략하고 객체의 속성 중 가장 중요한 것에만 중점을 둬 개략화(모델화) 하는 것이다.

- 데이터의 공통된 성질을 추출하여 슈퍼 클래스를 선정하는 개념이다.

3. **상속성** : 이미 정의된 상위 클래스(부모 클래스)의 모든 속성과 연산을 하위 클래스가 물려받는 것이다.

- 싱속성을 이용하면 하위 클래스는 상위 클래스의 모든 속성과 연산을 자신의 클래스 내에서 다시 정의하지 않고 즉시 자신의 속성으로 이용 가능하다.
- 종류로 단일 상속과 다중 상속이 있다.

4. **다형성** : 메시지에 의해 객체(클래스)가 연산을 수행할 때 하나의 메시지에 대해 개체마다 고유한 방법으로 응답할 수 있는 능력이다.

- 객체는 동일한 메서드명을 사용하며 같은 의미의 응답을 한다.
- 같은 이름을 가진 기능이 상황에 따라 다르게 동작하는 성질을 말한다.
- 코드를 유연하고 재사용성을 높게 만든다.

 # 객체 지향 프로그래밍 원칙

 1. **단일 책임 원칙(SRP)**

객체는 오직 하나의 책임을 가져야 한다(객체는 오직 하나의 변경의 이유만을 가져야 한다).
- 이처럼 단일 책임 원칙은 클래스의 목적을 명확히 함으로써 구조가 난잡해지거나 수정 사항이 불필요하게 넓게 퍼지는 것을 예방하고 기능을 명확히 분리할 수 있게 한다.

2. **개방 폐쇄 원칙**(SRP)

객체는 확장에 대해서는 개방적이고 수정에 대해서는 폐쇄적이어야 한다는 원칙이다. 즉, 객체 기능의 확장을 허용하고 스스로의 변경은 피해야 한다.
- 예) 기존 앱에 새로운 기능 추가 → 앱 전체를 다시 만들 필요 없이, 플러그인이나 새로운 모듈만 추가
- 예) 기존 기능은 그대로 유지 → 사용자에게 안정성 제공

3. **리스코프 치환 원칙**(LSP)

자식 클래스는 언제나 자신의 부모 클래스를 대체할 수 있다는 원칙이다. 
- 예) 부모 클래스 → 콘센트
- 자식 클래스 → 전기기구 (예: 전구, 선풍기)
- 콘센트에 꽂는 코드를 바꾸지 않고도 전구든 선풍기든 문제없이 작동해야 한다

4. **인터페이스 분리 원칙**(ISP)

클라이언트에서 사용하지 않는 메서드는 사용해선 안 된다. 그러므로 인터페이스를 다시 작게 나누어 만든다.

- 클래스는 자신이 사용하지 않는 기능까지 강제로 구현하지 않도록, 인터페이스를 작게 나눠서 사용해야 한다.

5. **의존성 역전 원칙**(DIP)

추상성이 높고 안정적인 고수준의 클래스는 구체적이고 불안정한 저수준의 클래스에 의존해서는 안 된다는 원칙이다.

상위(추상적) 클래스는 하위(구체적) 클래스에 직접 의존하면 안 되고, 대신 추상적인 인터페이스나 클래스에 의존해야 한다.

- 예) 비유: 전기 코드와 가전제품

- 상위 수준 : 전기 콘센트 → 전원을 공급하는 핵심 역할

- 하위 수준 : 전구, 선풍기 등 → 세부적인 장치

- 전기 콘센트가 “전구”만을 위해 만들어져 있으면 안 된다.

- 인터페이스(전원 공급 규격)에 의존한다.

- 나중에 전구 대신 선풍기, 히터를 꽂아도 콘센트 코드는 그대로 사용할 수 있다.

장점으로는

- 상속을 통해 재사용과 시스템 확장이 용이하다.
- 코드의 재활용성이 높다.
- 자연적인 모델링에 의해 분석과 설계를 쉽고 효율적으로 할 수 있다.
- 사용자와 개발자 사이의 이해가 쉽다.
- 대형 프로그램 작성에 좋다.
- 소프트웨어 개발 및 유지보수가 좋다.

단점으로는

- 프로그래밍 구현을 지원해주는 정형화된 분석 및 설계방법이 없다.
- 구현 시 처리 시간이 지연된다.
